## プロジェクト概要

### 前提
- 実機は、車輪で移動する低背な台車ロボットである
- 台車ロボットは、全方位移動、円弧走行、その場回転などを行える
- ロボットにLiDARを設置し、3D SLAMで地面の凹凸、障害物、天井の様子を検知する
- ロボットが通行できない穴、凸凹、溝などの場所を検知する
- 検知結果をリアルタイムに扱うか、点群マップ上の位置として扱うかは未定だが、まずは点群マップ作成を目指す
- 坂道を含む、平坦ではない環境を対象とする
- LiDARは1台または2台程度を想定し、検知に必要な視野を確保できる台数と設置位置を検討する

### シミュレーション
- シミュレーションにはGazeboを使用する
- Gazebo上でLiDARの設置位置を調整し、このワークスペースで最適な設置位置を検討する
- 3D SLAMを検証するため、外部からロボットを操作できる機能を用意する
- 実機の動きを模擬できればよいため、シミュレーションでは必ずしも車輪機構を厳密に再現しなくてよい

### 本番環境
- ロボットのOSは、Jetson Orin NX上のJetPack 6を想定する
- 本番環境ではDockerを使わず、必要なソフトウェアを直接インストールする
- 開発環境では、本番環境にできるだけ近いDocker環境を構築する
- GPUは基本的に使用しないため、開発環境はCPUのみでよい
- 開発環境のDockerコンテナは、本番環境への直接インストールを想定して、インストールスクリプトや起動スクリプトを作成する
- 本番環境で使用するソース・スクリプトからは、シミュレーションや開発環境を意識しない作りにする

### その他
- 要件を満たす既存ライブラリがあれば利用してよい
- 有名で実績のあるライブラリがある場合は、積極的に採用を検討する
- 少しずつ検証しながら開発を進める
- 調査等で必要になる場合、調査用のツールやスクリプトをdev/agent_toolsディレクトリ内に作成してよい。その際、使用目的をdev/agent_tools/README.mdに記載すること
- ros2 daemonがエラーを出しやすい、下記のようなgraph確認系のros2コマンドを実行する際は--no-daemonオプションを付加すること
  - topic list
  - node list
  - service list
  - action list
  - topic info
  - node info
  - service info
  - action info

### フォルダ構成の概要
- `ros2_ws/`: ros2パッケージのワークスペース
- `scripts/`: 運用スクリプト群

## Python実行環境
- Pythonを実行する際は必ずシステム環境にインストールされたros2 humbleのPythonを使用すること
- システム既定のPythonを使用することは禁止する

## C++実行環境
- C++コードのビルド・実行は、必ずプロジェクトで定義された開発環境内で行うこと
- グローバル環境や個人環境にだけ依存する手順は禁止すること
- 依存関係を追加した場合は、ビルド設定・依存定義・実行手順を必ず更新すること
- ビルド、テスト、実行確認は再現可能なコマンドで行い、完了報告に実行したコマンドを記載すること
- 生成物、キャッシュ、一時ファイルはソース管理対象に含めないこと

## アプローチ
1. 言語ごとの標準的な書き方に従う
   - Python: PEP 8 と Python のイディオムに従う
   - C++: C++17以上を基本とし、RAII、所有権、ライフタイムを明確にする
2. 継承よりもコンポジションを優先する
3. 不要なメモリ確保・コピーを避ける
   - Python: 必要に応じてジェネレータを使用する
   - C++: `const&`、ムーブ、事前確保、スマートポインタを適切に使う
4. カスタム例外や戻り値を使用した包括的なエラー処理
   - エラーは握りつぶさず、例外、戻り値、ログを用途に応じて使い分ける
5. エッジケースを含む 90% 以上のテストカバレッジ
6. グローバル状態や暗黙の副作用を避け、テストしやすい小さな単位に分割する
7. 並行処理ではデータ競合、ライフタイム、ロック範囲を明確にする

## 注意点
- 型ヒントを用いた Python コードのクリーン化
- ドキュメント文字列とサンプルによるドキュメント化
- Python の標準ライブラリを活用し、サードパーティ製パッケージは慎重に使用すること
- 生成されたコードを自己レビューすること
- コードを生成・修正する際は、自明でない処理ブロックごとに、日本語コメントを必ず1行入れること。
- コメントは各ブロックの先頭に置き、5〜10行程度の処理まとまりに対して1つを目安とする。
- コメントでは処理の目的・意図・責務を説明すること。コードの単純な言い換えは禁止する。
- setter/getter、単純な代入、名前から意味が明白な1行処理など、自明な箇所にはコメントを書かないこと。
- コメントが不足している実装は未完了として扱い、完了報告前に自己修正すること。
- 思考は英語で行い、ユーザーへのレスポンスは日本語に翻訳すること
- アーティファクトはユーザーにレビューを求める前に日本語に翻訳すること
- タスクを実行中、Stepが10を超えたら一度処理を停止して「完了した作業」「状況の報告」「次のアクション」を報告し、ユーザーの判断を仰ぐこと
- 外部リポジトリのソースに対しパッチを当てる場合は、できる限り元のソースを活かし、改修量が少なくなるようにすること。また、改修箇所をコメントで囲い、わかりやすくすること
- 明示的に指示した場合を除き、既存機能の互換は不要とする。完全に新しい機能に切り替え、不要となったコードは削除すること

## 開発ワークフロー

### 新機能追加時
1. 調査
2. 計画（.kilo/plansディレクトリに.mdファイルを出力すること）
3. 実装（.mdファイルに基づき実装すること）
4. テスト
5. コミット

### バグ修正時
1. 原因調査
2. テスト作成（TDD）
3. 修正実装
4. 動作確認
